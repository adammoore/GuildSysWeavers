This document describes how decisions are made within the Guild of Systems Weavers.

## Scope of decisions

Decisions fall into three categories:

### Content decisions

Adding, modifying, or removing patterns, case studies, glossary entries, and other knowledge content.

**Process:** Pull request review by at least one maintainer. For substantive new content (e.g. a new pattern), discussion in an issue first is encouraged.

### Structural decisions

Changes to the repository structure, site organisation, templates, or contribution processes.

**Process:** Proposal via issue, discussion period of at least one week, implementation via pull request.

### Governance decisions

Changes to governance, recognition pathways, the charter, or other foundational elements.

**Process:** Proposal via issue, discussion period of at least two weeks, broad input sought from active contributors, implementation via pull request.

## Consensus and disagreement

The guild aims for rough consensus. Where disagreement exists:

1. The disagreement is documented in the issue
2. Participants are asked to articulate their concerns clearly
3. Maintainers seek a resolution that addresses the core concerns
4. If consensus cannot be reached, maintainers may make a decision and document the rationale and dissenting views

## Decision records

Significant decisions are recorded in the repository with their rationale, so that future contributors can understand why things are the way they are.

## Evolution

This model is itself subject to change as the community grows. It is designed for the current founding phase and should be revisited as the guild develops.
